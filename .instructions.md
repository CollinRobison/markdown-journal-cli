# Markdown Journal CLI - Development Instructions

## Project Overview
A .NET 9 CLI application for creating and managing markdown-based journals with automatic table of contents, file tracking, and template-driven content generation. Built with Spectre.Console.Cli framework for rich terminal UI and structured command architecture.

## Architectural Patterns

### 1. Command Pattern (Spectre.Console.Cli)
**Pattern**: Commands are implemented using Spectre.Console.Cli's command framework, separating CLI concerns from business logic.

**Core Principle - Thin Command Delegation**:
Commands should be THIN and delegate ALL business logic to services. A well-designed command:
- Contains ONLY validation, service delegation, and UI output
- Should be under 100 lines of code (excluding comments)
- Has NO static helper methods or embedded business logic
- Orchestrates service calls but does not implement domain operations
- Follows the delegation pattern demonstrated by `AddEntryCommand`

**Anti-Pattern Examples (AVOID)**:
```csharp
// ❌ BAD: Command contains business logic and static helpers
public class BadUpdateCommand : Command<UpdateSettings>
{
    public override int Execute(...)
    {
        // Direct file manipulation in command
        File.Copy(oldPath, newPath);
        
        // Inline config manipulation
        var config = JsonSerializer.Deserialize<Config>(json);
        config.Entries = UpdateEntries(config.Entries);
        
        // Many operations orchestrated here
        // ... 200+ lines of code
    }
    
    private static void UpdateEntries(...) { /* business logic */ }
    private static Entry FindEntry(...) { /* business logic */ }
}
```

**Good Pattern (FOLLOW THIS)**:
```csharp
// ✅ GOOD: Command delegates to services
public class GoodUpdateCommand(
    IAnsiConsole console,
    IJournalFileUpdateService fileUpdateService,
    ITableOfContentsService tocService
) : Command<UpdateSettings>
{
    public override int Execute(CommandContext context, UpdateSettings settings)
    {
        try
        {
            // Simple validation
            if (string.IsNullOrEmpty(settings.FileName))
            {
                _console.MarkupLine("[red]Error:[/] File name required.");
                return 1;
            }
            
            // Delegate to service
            _fileUpdateService.RenameEntry(directory, oldFile, newFile);
            _tocService.UpdateTableOfContents(directory);
            
            // Simple output
            _console.MarkupLine("[green]Success:[/] Entry updated.");
            return 0;
        }
        catch (JournalException ex)
        {
            _console.MarkupLine($"[red]Error:[/] {ex.Message}");
            return 1;
        }
    }
}
```

**When to Create a New Service**:
- Command exceeds ~80 lines of business logic
- You're tempted to add static helper methods to a command
- Multiple commands need the same operation (e.g., finding entries, updating files)
- Logic involves complex orchestration (file rename + config update + tracking)

**Implementation**:
- All commands inherit from `Command<TSettings>` where `TSettings : CommandSettings`
- Commands are registered in `Program.cs` using `config.AddCommand<T>()` or `config.AddBranch<T>()`
- Commands receive dependencies via constructor injection
- Commands focus ONLY on CLI input/output; delegate business logic to services

**Structure**:
```csharp
public sealed class YourCommand(
    IAnsiConsole console,
    IServiceOne service1,
    IServiceTwo service2
) : Command<YourCommandSettings>
{
    public override int Execute(CommandContext context, YourCommandSettings settings)
    {
        try
        {
            // 1. Call service methods to perform business logic
            // 2. Use _console.MarkupLine() for output (Spectre markup)
            // 3. Return 0 for success, 1 for errors
        }
        catch (SpecificJournalException ex)
        {
            _console.MarkupLine($"[red]Error:[/] {ex.Message}");
            return 1;
        }
    }
}

public sealed class YourCommandSettings : CommandSettings
{
    [CommandArgument(0, "[name]")]
    [Description("Description of the argument")]
    public string? Name { get; init; }

    [CommandOption("-p|--path")]
    [Description("Description of the option")]
    public string? Path { get; init; }

    public override ValidationResult Validate()
    {
        // Validate inputs before command execution
        return ValidationResult.Success();
    }
}
```

**Command Registration**:
```csharp
// Single command
config.AddCommand<NewCommand>("new");

// Command branch (grouped subcommands like 'add entry', 'add config')
config.AddBranch<AddSettings>("add", add =>
{
    add.AddCommand<AddEntry>("entry");
    add.AddCommand<AddJournalrc>("config");
});
```

### 2. Service-Oriented Architecture
**Pattern**: Business logic is encapsulated in services, following Interface Segregation and Single Responsibility principles.

**Service Layer Structure**:
- **Commands/** - CLI interface layer (thin, delegates to services)
- **Services/** - High-level business logic orchestration
- **Infrastructure/** - Core technical services and abstractions

**Service Design**:
- All services MUST have an interface (e.g., `IFileSystem`, `IJournalConfiguration`)
- Services are registered as singletons in `Program.cs`
- Services receive dependencies via constructor injection
- Constructor parameters MUST validate for null and throw `ArgumentNullException`

**Example Service Implementation**:
```csharp
public class JournalEntryService(
    IFileSystem fileSystem,
    IJournalConfiguration journalConfiguration,
    ITemplateManager templateManager,
    ILogger<JournalEntryService> logger
) : IJournalEntryService
{
    private readonly IFileSystem _fileSystem =
        fileSystem ?? throw new ArgumentNullException(nameof(fileSystem));
    // ... other null checks

    public void AddEntry(string path, string name)
    {
        _logger.LogDebug("Adding entry '{Name}' to '{Path}'", name, path);
        
        // Business logic implementation
        var content = _templateManager.GenerateFromTemplate("journal-entry", parameters);
        _fileSystem.WriteAllText(fullPath, content);
    }
}
```

**Service Registration** (in Program.cs):
```csharp
// Infrastructure services
host.Services.AddSingleton<IFileSystem, FileSystem>();
host.Services.AddSingleton<ITemplateManager, TemplateManager>();

// Business services
host.Services.AddSingleton<IJournalEntryService, JournalEntryService>();

// Commands
host.Services.AddSingleton<AddEntry>();
```

### 3. Dependency Injection - Adapter Pattern
**Pattern**: `TypeRegistrar` acts as an adapter between Spectre.Console.Cli's DI abstractions and Microsoft.Extensions.DependencyInjection.

**Implementation**:
- `TypeRegistrar` implements `ITypeRegistrar` (Spectre)
- Wraps `IServiceCollection` (Microsoft DI)
- `TypeResolver` implements `ITypeResolver`, wraps `IServiceProvider`

**Why**: Spectre.Console.Cli is DI-framework agnostic; this adapter lets us use Microsoft's powerful DI container while maintaining compatibility.

**Location**: `markdown_journal_cli.Infrastructure.DependencyInjection.TypeRegistrar`

### 4. Strategy Pattern - Template Generators
**Pattern**: Template generators implement a common interface, allowing runtime selection and registration of different template types.

**Implementation**:
```csharp
public interface ITemplateGenerator
{
    string TemplateName { get; }
    string GenerateTemplate(Dictionary<string, object>? parameters);
}

public class JournalEntryTemplate : ITemplateGenerator
{
    public string TemplateName => "journal-entry";
    
    public string GenerateTemplate(Dictionary<string, object>? parameters)
    {
        var title = parameters?.GetValueOrDefault("title", "Default Title");
        return $"# {title}\n\n...";
    }
}
```

**Template Manager** (Registry Pattern):
```csharp
public interface ITemplateManager
{
    void RegisterTemplate(ITemplateGenerator template);
    string GenerateFromTemplate(string templateName, Dictionary<string, object>? parameters);
    IEnumerable<string> GetAvailableTemplates();
}
```

**Usage**:
- Templates are registered in `TemplateManager.RegisterDefaultTemplates()`
- Generate content: `_templateManager.GenerateFromTemplate("journal-entry", parameters)`
- Parameters are passed as `Dictionary<string, object>` for flexibility

**Location**: `markdown_journal_cli.Infrastructure.JournalTemplates`

### 5. Repository Pattern - File System Abstraction
**Pattern**: File system operations are abstracted behind `IFileSystem` interface for testability and cross-platform compatibility.

**Implementation**:
- Production: `FileSystem` wraps `System.IO` operations
- Testing: Mock or custom test implementation
- Benefits: Unit tests don't touch real file system

**Interface**:
```csharp
public interface IFileSystem
{
    bool FileExists(string path);
    void WriteAllText(string path, string content);
    string ReadAllText(string path);
    void CreateDirectory(string path);
    // ... other file operations
}
```

## Spectre.Console Usage

### When to Invoke the Spectre.Console Skill
**ALWAYS use the `spectre-console` skill when**:
- Creating or modifying command UI (tables, prompts, progress bars)
- Implementing interactive user input (selections, confirmations)
- Adding colorized or styled console output
- Creating status indicators or progress tracking
- Working with Spectre.Console.Cli command framework

**How to invoke**:
When working on commands, UI rendering, or interactive features, reference the [spectre-console skill](.github/skills/spectre-console/SKILL.md) for best practices and patterns.

### Spectre.Console Standards
- Use `IAnsiConsole.MarkupLine()` for all console output (injected via DI)
- Use Spectre markup tags: `[green]`, `[red]`, `[bold]`, `[underline]`
- Call `.EscapeMarkup()` on user input before interpolation
- Use `SelectionPrompt<T>` and `MultiSelectionPrompt<T>` for selections
- Use `AnsiConsole.Status()` for short operations, `AnsiConsole.Progress()` for long ones
- Inject `IAnsiConsole` into commands for testability

**Example**:
```csharp
public sealed class AddEntry(IAnsiConsole console, IJournalEntryService service) : Command<AddEntrySettings>
{
    public override int Execute(CommandContext context, AddEntrySettings settings)
    {
        try
        {
            service.AddEntry(settings.Path, settings.EntryName);
            _console.MarkupLine("[green]✓[/] Entry created successfully");
            return 0;
        }
        catch (JournalException ex)
        {
            _console.MarkupLine($"[red]Error:[/] {ex.Message}");
            return 1;
        }
    }
}
```

## Exception Handling

### Exception Hierarchy
```
System.Exception
    └── JournalException (Base for all journal-specific errors)
            ├── JournalrcNotFoundException
            ├── JournalEntryAlreadyExistsException
            ├── TrackingIndexNotFoundException
            └── [Future exceptions inherit from JournalException]
```

### Exception Best Practices
- Create specific exception types for different error scenarios
- Inherit from `JournalException` base class
- Provide constructor chaining: `base(message)` and `base(message, innerException)`
- Include contextual information in exception messages
- Catch specific exceptions in commands before generic ones

**Exception Implementation**:
```csharp
public class YourSpecificException : JournalException
{
    public YourSpecificException(string message) : base(message) { }
    
    public YourSpecificException(string message, Exception innerException) 
        : base(message, innerException) { }
}
```

**Command Exception Handling**:
```csharp
try
{
    // Command logic
}
catch (JournalrcNotFoundException ex)  // Most specific first
{
    _console.MarkupLine($"[red]Error:[/] {ex.Message}");
    return 1;
}
catch (JournalException ex)  // Base journal exception
{
    _console.MarkupLine($"[red]Error:[/] {ex.Message}");
    return 1;
}
catch (Exception ex)  // Unexpected errors
{
    _console.MarkupLine($"[red]Error:[/] An unexpected error occurred: {ex.Message}");
    return 1;
}
```

**Location**: `markdown_journal_cli.Exceptions.JournalExceptions`

## Testing Requirements

### Testing Framework Stack
- **xUnit** - Test framework
- **Moq** - Mocking library
- **Shouldly** - Assertion library (fluent syntax)

### Test Structure
Every new service, command, or component MUST have corresponding unit tests in `markdown-journal-cli.Tests/` mirroring the source structure:

```
markdown-journal-cli/Commands/Add/AddEntryCommand.cs
→ markdown-journal-cli.Tests/Commands/Add/AddEntryCommandTests.cs

markdown-journal-cli/Services/JournalEntryService.cs
→ markdown-journal-cli.Tests/Services/JournalEntryServiceTests.cs
```

### Test Class Template
```csharp
using Moq;
using Shouldly;
using Xunit;

namespace markdown_journal_cli.Tests.Services;

/// <summary>
/// Unit tests for <see cref="YourService"/> covering all public behavior.
/// </summary>
public class YourServiceTests
{
    private readonly Mock<IDependency1> _mockDependency1;
    private readonly Mock<IDependency2> _mockDependency2;
    private readonly YourService _service;

    public YourServiceTests()
    {
        _mockDependency1 = new Mock<IDependency1>();
        _mockDependency2 = new Mock<IDependency2>();
        
        _service = new YourService(
            _mockDependency1.Object,
            _mockDependency2.Object
        );
    }

    [Fact]
    public void MethodName_Should_ExpectedBehavior_When_Condition()
    {
        // Given (Arrange)
        var input = "test";
        _mockDependency1.Setup(x => x.DoSomething(input)).Returns("result");

        // When (Act)
        var result = _service.MethodName(input);

        // Then (Assert)
        result.ShouldBe("expected");
        _mockDependency1.Verify(x => x.DoSomething(input), Times.Once);
    }

    [Fact]
    public void MethodName_Should_ThrowException_When_InvalidInput()
    {
        // Given
        var invalidInput = string.Empty;

        // When / Then
        Should.Throw<ArgumentException>(() => _service.MethodName(invalidInput));
    }
}
```

### Test Naming Convention
Use the pattern: `MethodName_Should_ExpectedBehavior_When_Condition`

**Examples**:
- `AddEntry_Should_CreateFile_When_ValidParameters`
- `AddEntry_Should_ThrowException_When_JournalrcNotFound`
- `GenerateTemplate_Should_UseDefaults_When_ParametersNull`

### Test Coverage Requirements
- Test happy path (expected behavior with valid inputs)
- Test edge cases (null, empty, boundary values)
- Test exception scenarios
- Verify service method calls with `Mock.Verify()`
- Use `Shouldly` assertions for readability

### Assertion Examples with Shouldly
```csharp
result.ShouldBe("expected");
result.ShouldContain("substring");
result.ShouldNotBeNull();
exception.Message.ShouldContain("error message");
list.ShouldBeEmpty();
_mockService.Verify(x => x.Method(), Times.Once);
```

## File Organization and Naming

### Directory Structure
- **Commands/** - Grouped by command (Add/, New/, Update/)
- **Services/** - High-level business logic
- **Infrastructure/** - Core technical services
  - **Configuration/** - Journal config management
  - **DependencyInjection/** - DI adapters
  - **FileSystem/** - File operations
  - **JournalTemplates/** - Template system
  - **Tracking/** - File change detection

### Naming Conventions
- **Interfaces**: Prefix with `I` (e.g., `IFileSystem`, `IJournalConfiguration`)
- **Implementations**: Match interface name without `I` (e.g., `FileSystem`, `JournalConfiguration`)
- **Commands**: Suffix with `Command` (e.g., `AddEntryCommand`, `NewCommand`)
- **Settings**: Suffix with `Settings` (e.g., `AddEntrySettings`, `NewCommandSettings`)
- **Tests**: Suffix with `Tests` (e.g., `FileSystemTests`, `AddEntryCommandTests`)
- **Services**: Suffix with `Service` (e.g., `JournalEntryService`, `EntryFormatterService`)

### Code Organization
- One class per file
- File name matches class name exactly
- Nested classes (like `NewCommand.Settings`) can be in the same file as parent
- Group related files in subdirectories

## Development Workflow

### Adding a New Command
1. Create command class in `markdown-journal-cli/Commands/{CommandGroup}/`
2. Create settings class (inherit from `CommandSettings`)
3. Inject required services via constructor
4. Implement `Execute()` method
5. Register command in `Program.cs`
6. Create corresponding test class in `markdown-journal-cli.Tests/Commands/{CommandGroup}/`
7. Write unit tests covering all scenarios

### Adding a New Service
1. Create interface in appropriate `Infrastructure/` or `Services/` folder
2. Create implementation class
3. Register service in `Program.cs` as singleton
4. Create corresponding test class
5. Write unit tests with mocked dependencies

### Adding a New Template
1. Create class implementing `ITemplateGenerator` in `Infrastructure/JournalTemplates/Templates/`
2. Implement `TemplateName` property and `GenerateTemplate()` method
3. Register in `TemplateManager.RegisterDefaultTemplates()`
4. Create test class in `markdown-journal-cli.Tests/Infrastructure/JournalTemplates/Templates/`
5. Test all parameter combinations and edge cases

### Modifying Existing Code
1. Read existing tests to understand expected behavior
2. Write new tests for the desired behavior (TDD approach)
3. Make implementation changes
4. Run `dotnet test` to ensure all tests pass
5. Update documentation if architectural changes made

## Logging
- Use `ILogger<T>` for structured logging (injected via DI)
- Log at appropriate levels: Debug, Information, Warning, Error
- Include contextual information in log messages using structured logging syntax

**Example**:
```csharp
_logger.LogDebug("Adding entry '{EntryName}' to '{Path}'", entryName, path);
_logger.LogWarning("Journal config not found at '{FilePath}'", filePath);
```

## Documentation Standards
- Use XML documentation comments (`///`) for all public interfaces and classes
- Include `<summary>`, `<param>`, `<returns>`, and `<exception>` tags
- Keep documentation concise and focused on intent, not implementation
- Update README.md and docs/ARCHITECTURE.md for significant architectural changes

## Code Style
- Use primary constructors with parameter validation
- Initialize readonly fields from constructor parameters
- Use expression-bodied members where appropriate
- Use file-scoped namespaces (`namespace X;`)
- Prefer `readonly` fields over mutable state
- Use `sealed` for classes that shouldn't be inherited
- Use nullable reference types (`string?` for nullable, `string` for non-null)

## Anti-Patterns to Avoid
- ❌ Don't put business logic in commands (delegate to services)
- ❌ Don't directly instantiate services in constructors (`new FileSystem()`)
- ❌ Don't use `Console.WriteLine()` (use `IAnsiConsole.MarkupLine()`)
- ❌ Don't forget to escape user input before using in Spectre markup
- ❌ Don't create services without interfaces
- ❌ Don't forget constructor null checks for injected dependencies
- ❌ Don't catch generic `Exception` before specific exceptions
- ❌ Don't create production code without corresponding unit tests
